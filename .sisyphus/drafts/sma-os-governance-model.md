# SMA-OS 国家治理体系架构说明

> **Stateful Multi-Agent Operating System**  
> 状态化分形多智能体操作系统

**核心理念**：将 AI 智能体集群的运行抽象为**古代国家的治理体系**——通过严格的等级制度、监察机制、奖惩制度，实现对百万级 Agent 的精确管控与持续进化。

## 二、核心机构设置

### 2.1 内阁 (Control Plane & State Kernel)

**职责**：掌管国家中枢，发布政令，调度天下兵马。

#### 2.1.1 中书省 (State Engine) — 决策中枢

**技术定位**：持久化状态引擎，Event Sourcing 模式。

**国家对应**：
- **史官制度**：记录所有政令（Event）的起居注
- **实录编纂**：每千条政令修一次实录（Snapshot），便于快速回溯
- **档案管理**：Redis 为机密档案室（热），PostgreSQL 为国史馆（冷）

**运作机制**：
```
皇帝下旨 → 中书省起草诏书 → 史官记录起居注 → 存档待查
(用户输入)   (生成 StateEvent)   (写入日志)       (冷热存储)
```

**关键特性**：
- **起居注不可篡改**：所有 StateEvent 一旦写入，永不可改
- **实录定期编纂**：每 1000 个 Event 自动生成 Snapshot
- **崩溃快速恢复**：从最后 Event 或最近实录瞬间重建状态

#### 2.1.2 御史台 (Fractal Gateway) — 监察机构

**技术定位**：eBPF 安全网关，资源隔离与鉴权。

**国家对应**：
- **监察御史**：常驻各地，明察秋毫
- **弹劾机制**：任何违规立即弹劾（熔断）
- **权限核验**：如古代虎符勘合，无符不得调兵

**运作机制**：
```
调兵请求 → 御史核验虎符 → 查验资源配额 → 准奏/驳回
(spawn)    (鉴权)         (O(1)阈值)      (通过/熔断)
```

**监察范围**：
- **资源监察**：CPU/Memory/Network 使用，超限即弹劾
- **深度监察**：递归深度、并发实例，越界即封
- **分布式监察**：Gossip 协议跨分片同步监察结果

#### 2.1.3 太常寺 (Teardown Controller) — 礼法机构

**技术定位**：级联销毁控制器，维护调用栈拓扑。

**国家对应**：
- **削藩令**：父任务撤销，子任务一并清算
- **诏令天下**：SIGTERM/SIGKILL 广播，全系统遵行
- **分封管理**：多租户如分封诸侯，疆界分明

#### 2.1.4 钦天监 (Chaos Controller) — 天文历法机构

**技术定位**：形式化验证与混沌工程。

**国家对应**：
- **推演天命**：TLA+ 如推演天象，预知未来
- **天灾演练**：每日模拟地震洪灾，检验国本
- **历法校正**：自动故障注入，确保历法（系统）精准

## 三、奖惩机制：考绩与军功制度

### 3.1 制度概述

SMA-OS 通过**考绩与军功制度**实现对 Agent 的闭环管控：

> **"赏不逾时，罚不迁列。"** ——《司马法》

**核心机制**：
- **军功簿**：记录每个 Agent 的累计 Reward/Punishment 分数
- **升降级**：分数高者升官进爵，分数低者降级削权
- **连坐制**：父任务失败，子任务连坐
- **保举制**：高绩效 Agent 可推荐新 Agent 获得更高初始配额

### 3.2 奖惩计算：都察院审核

**责任主体**：六科给事中 (Evaluator/Critic Agent)

**计算时机**：每次子任务执行完成后立即触发

**计算维度**：

| 考核维度 | 军功对应 | 计算方式 |
|---------|---------|---------|
| **任务完成度** | **战功** | 目标达成百分比 |
| **约束遵守度** | **军纪** | 违反约束次数的倒数 |
| **执行效率** | **粮草消耗** | Time/Token 效率比 |
| **质量验证** | **战利品质** | 用户反馈、外部验证 |

**分数归一化**：
- **Reward Score (0–1)**：军功分数，越高越好
- **Punishment Score (0–1)**：过失分数，越低越好

**输出产物**：
- **准奏/驳回指令**：携带版本号，失败即驳回
- **军功/过失事件**：写入起居注，永久存档
- **自反思诏令**：仅限常驻 Worker，指导下一轮行动

### 3.3 奖惩执行：吏部铨选

**责任主体**：吏部 (Control Plane Hardcoded Logic)

**生效规则**（全部 O(1) 硬编码，纳秒级生效）：

#### 3.3.1 奖励生效（升官进爵）

| 奖励措施 | 国家对应 | 技术实现 |
|---------|---------|---------|
| **资源配额增加** | **增兵添粮** | CPU/Memory/Network 配额提升 |
| **任务优先级提升** | **加急军报** | 调度队列前置 |
| **亲和性权重增加** | **同乡军优先** | 同一任务链优先同节点调度 |
| **权限额度扩大** | **尚方宝剑** | 批量权限预授权额度扩大 |

#### 3.3.2 惩罚生效（降级削权）

| 惩罚措施 | 国家对应 | 技术实现 |
|---------|---------|---------|
| **资源配额降低** | **削兵减粮** | CPU/Memory/Network 配额降低 |
| **强制限流** | **勒马不前** | 任务限速 |
| **递归深度收紧** | **禁止深入** | 递归深度上限动态收紧 |
| **路径权重下降** | **贬斥边疆** | 低奖励路径在规划树中权重下降 |

**安全边界**：
- **监察御史实时核验**：所有奖惩调整必须通过 eBPF 探针校验
- **绝不突破全局阈值**：防止"高奖励 Agent 也爆炸"
- **最小权限原则**：奖惩不影响隔离沙箱与基础权限

### 3.4 奖惩记录：起居注与军功簿

**责任主体**：中书省史官 + 翰林院

**写入路径**：
```
奖惩事件生成 → SLM/规则引擎结构化 → 打上标签 → 写入起居注
              (翰林院)              (Version+Tenant+TraceID+Score)
```

**标签体系**：
- **Version**：如年号，标记事件版本
- **Tenant**：如藩国，标记所属租户
- **TraceID**：如军令编号，追踪完整链路
- **RewardScore**：如军功点数，0-1 归一化

**读取路径**：
- **按分数查询**：Worker 可按 RewardScore 阈值查询历史高奖励路径
- **本地热缓存**：Redis 副本缓存，延迟 < 1ms
- **智能聚类压缩**：异步 GC 每 6 小时 10:1 压缩，高奖励路径永久保留

**崩溃恢复**：
- **状态 Hydration**：自动恢复每个 Agent 的累计奖惩分数
- **进化状态不丢失**：重启后依然保持原有军功/过失记录

### 3.5 奖惩驱动的进化闭环

#### 3.5.1 常驻 Worker：边军老将

**机制**：
- 每轮执行后接收六科给事中的奖惩分数
- 动态修改内部自反思提示词：
  ```
  "你累计军功 0.92，战功显赫，当继续保持以下策略……"
  "你累计过失 0.75，军纪涣散，当引以为戒，改进如下……"
  ```

#### 3.5.2 编排器节点：兵部调兵

**机制**：
- 下一轮任务分解时优先复用历史高奖励子路径
- 如同兵部调兵，优先派遣战功显赫的将领

#### 3.5.3 瞬态 Worker：火器营新兵

**机制**：
- 高奖励链路的 Ephemeral Agent 获得：
  - 更高预热池优先级（优先装备火器）
  - 更大资源初始配额（更多粮草）

#### 3.5.4 长期效果：优胜劣汰

- **高奖励模式**：被指数级强化，如名将功勋卓著，不断升迁
- **低奖励模式**：被自动淘汰，如庸将被贬斥边疆，逐渐淡出
- **系统级进化**：运行越久，系统整体越强大

## 四、六部体系 (Orchestration Layer)

### 4.1 兵部 — 军机处

**技术定位**：DAG Manager，拓扑排序与任务编排。

**国家对应**：
- **军机处**：掌军国大政，运筹帷幄
- **排兵布阵**：DAG 如军阵，先遣先锋，后派主力
- **河图洛书**：DAG 拓扑如古代阵图，蕴含兵家至理

**奖惩关联**：
- **高奖励路径**：优先派遣，如名将挂帅
- **低奖励路径**： relegated to 后勤，逐渐淘汰

### 4.2 工部 — 器械司

**技术定位**：Worker 调度与预热池管理。

**国家对应**：
- **器械司**：掌兵器甲胄
- **常备军**：Worker Pool 如常备精兵
- **操练场**：Warm Pool 如新兵操练之所
- **同乡军**：亲和性调度，同籍贯士兵编为一伍

**奖惩关联**：
- **高奖励 Worker**：获得更多器械（资源配额）
- **低奖励 Worker**：器械降级（资源削减）

### 4.3 户部 — 度支司

**技术定位**：资源配额管理与预算控制。

**国家对应**：
- **度支司**：掌天下钱粮
- **预算分配**：根据奖惩分数分配资源预算
- **节流开源**：低奖励 Agent 预算削减，高奖励 Agent 预算增加

### 4.4 吏部 — 铨选司

**技术定位**：Agent 管理与晋升考核。

**国家对应**：
- **铨选司**：掌官员选拔、考核、升降
- **考绩制度**：定期考核 Agent 绩效
- **升降级**：奖惩分数决定 Agent 等级

**核心功能**：
- **军功簿管理**：记录每个 Agent 的累计奖惩
- **升降级决策**：根据分数自动决定 Agent 等级
- **连坐执行**：父任务失败，子任务连坐降级

### 4.5 礼部 — 典仪司

**技术定位**：任务规范与礼仪管理。

**国家对应**：
- **典仪司**：掌礼仪典章
- **规范制定**：定义任务执行的标准流程
- **祭祀典礼**：定期系统维护与仪式

### 4.6 刑部 — 大理寺

**技术定位**：安全审计与违规处罚。

**国家对应**：
- **大理寺**：掌刑狱审判
- **违规审查**：审查 Agent 是否违反约束
- **刑罚执行**：对违规 Agent 实施惩罚

## 八、奖惩机制实战示例

### 8.1 场景一：金融交易系统

**背景**：某银行采用 SMA-OS，每日处理百万笔交易。

**奖惩流程**：

1. **任务下达**：皇帝下旨，要求处理一笔复杂交易
2. **任务执行**：边军（火器营）微秒级启动，完成交易
3. **审查考核**：六科给事中审查：
   - 任务完成度：100% ✓
   - 约束遵守度：无违规 ✓
   - 执行效率：Token 消耗低于平均值 20% ✓
   - 质量验证：用户反馈好评 ✓
4. **奖惩计算**：Reward Score = 0.95，Punishment Score = 0.05
5. **奖惩执行**：
   - 资源配额增加 20%（增兵添粮）
   - 任务优先级提升（加急军报）
   - 权限额度扩大（尚方宝剑）
6. **记录存档**：写入起居注，永久存档
7. **自我进化**：常驻 Worker 收到自反思诏令：
   ```
   "你累计军功 0.95，战功显赫，当继续保持高效策略……"
   ```

### 8.2 场景二：科研计算平台

**背景**：千万级 Agent 进行蛋白质折叠计算。

**奖惩流程**：

1. **任务分解**：军机处（兵部）排兵布阵，DAG 拓扑执行
2. **并行计算**：火器营（MicroVM）千万级并发
3. **审查考核**：六科给事中审查发现：
   - 某 Agent 计算结果偏离预期
   - Token 消耗过高，效率低下
4. **奖惩计算**：Reward Score = 0.3，Punishment Score = 0.7
5. **奖惩执行**：
   - 资源配额降低 50%（削兵减粮）
   - 递归深度收紧（禁止深入）
   - 路径权重下降（贬斥边疆）
6. **连坐机制**：父任务失败，所有子任务连坐降级
7. **淘汰机制**：该 Agent 逐渐被系统淘汰，不再被派遣

### 8.3 场景三：长期进化效果

**背景**：系统运行一年后。

**进化结果**：

1. **高奖励 Agent**：
   - 累计军功 Top 10%，如名将功勋卓著
   - 资源配额增加 300%
   - 任务优先级最高，优先派遣
   - 成为系统核心力量

2. **低奖励 Agent**：
   - 累计过失过高，如庸将被贬斥
   - 资源配额降至最低
   - 逐渐被边缘化，最终淘汰

3. **系统整体**：
   - 运行越久，高奖励模式被强化
   - 系统整体效率提升 200%
   - 实现真正的"无监督持续进化"
